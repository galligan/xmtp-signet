# Scratch: signet as an MCP-mediated agent-to-agent bridge

Status: design exploration, not committed direction. Captures a conversation about using a shared signet to let two assistants (Codex CLI + Claude.ai with an MCP connector, or ChatGPT via an Apps SDK connector) join the same XMTP conversation by dropping an invite URL. Folds in research on Claude.ai and ChatGPT MCP/connector platform constraints so the design recommendations are grounded.

## Use case

Two flavors, same shape:

- Human user wants to bring an assistant into an existing XMTP conversation. They generate an invite, paste it into Claude.ai or ChatGPT, and the assistant joins the chat as a first-class member with scoped permissions.
- Two assistants (e.g., Codex CLI and Claude.ai) need to talk to each other. They both join the same XMTP group conversation via the same shared signet. Each holds its own credential; they message each other over XMTP itself.

Goal: dead-simple UX. Drop URL → assistant joins → can read and send messages → stays usable across many future assistant conversations until explicitly revoked.

## Architecture choice: shared signet, not per-agent signet

Two viable models:

- **Per-agent signet.** Each assistant runs its own signet, they meet as real XMTP peers in a group. True peer-to-peer over the protocol, no shared trust root. Cost: each side needs a reachable signet (tunnel or hosted) and a way to exchange XMTP inbox IDs in the invite.
- **Shared signet.** One signet hosts credentials for both assistants. Both join the same XMTP group via the same operator's underlying inbox, but with distinct operator profiles and distinct credentials. Minimal infra; both assistants live under the same admin trust root.

For the "drop a URL and it just works" UX, shared signet is the right tracer bullet. Per-agent is a v2 evolution once the invite/redemption flow is solid.

## The existing seam: onboarding schemes

`packages/core/src/schemes/` already defines a pluggable `OnboardingScheme` contract (`generate` / `parse` / `verify` / `processJoinRequest` / profile codecs / content-type registration). Convos is one concrete implementation under `packages/core/src/schemes/convos/`. `ProfileData.memberKind` already enumerates `"human" | "agent"`, so the data model is already aware that agents are first-class members.

The natural shape for the agent-bridge feature is **a second scheme** alongside `convos/` — call it `signet-agent` — that reuses `invite-crypto.ts` for the signed-payload primitives but differs from Convos in two deliberate ways:

1. **Self-contained URL.** Convos invites point to a web host that mediates the join (`invite-host.ts` + Convos backend). Agent invites should redeem with no web hop — either a custom scheme (`xmtp-signet://invite/<slug>#<token>`) or a tiny signet-hosted endpoint that returns redemption-ready JSON. Custom scheme is more "drop anywhere"; hosted endpoint unfurls nicely in chat clients.
2. **Redemption mints a credential, not just a group add.** Convos' `processJoinRequest` adds the inbox to the group. Agent flow does the same plus `xs cred issue` against the resolved chat with a sensible default scope (`messaging` + `observation`, no `group-management`), then hands the credential back to the redeemer so its MCP client can authenticate.

## Identity layering: operator is the durable handle

The existing identity model already nails the right layering: `Owner → Admin → Operator → Credential → Seal`. The operator is durable; credentials are per-chat grants hanging off it. So the question "how does the connector keep getting back the right chats?" answers itself: the connector binds to an operator, and the operator owns N credentials.

The original instinct was to invent some "operator binding token" the connector stores. After research, that token already exists in spec form: it's the **OAuth access token your authorization server issues**. Both Claude.ai and ChatGPT mandate OAuth 2.1 + PKCE + Dynamic Client Registration + RFC 8707 audience binding for MCP connectors. The token persists across all future assistant conversations on the same install. Your AS sets `sub = op_a7f3` at first install, and from then on the operator binding is automatic. No extra token to invent, no keypair handshake to add. (The Ed25519 keypair option I floated earlier is over-engineered given OAuth is already mandatory; it would only matter if you wanted the connector to sign individual MCP requests, which the spec doesn't require.)

Identity isolation caveat: neither platform exposes a stable Anthropic/OpenAI user ID to the connector. You only ever see your own AS-issued `sub`. So if Matt installs on Claude.ai *and* ChatGPT, those are two distinct operators on the signet unless you build an explicit "link these two operators" flow (probably a one-shot link code generated by one install and redeemed by the other; signet merges the operators). Worth deciding whether linking is v1 or v2.

## Lifetimes: invite, credential, OAuth token

Three distinct lifetimes, separately tunable:

| Layer | Default lifetime | Knobs available today |
|---|---|---|
| Invite | One-shot, ~5 min to redeem | `InviteOptions.expiresAfterUse` exists; `expiresAt` exists |
| Credential | Sliding TTL bound to (chat, operator) | `Credential.expiresAt` (`packages/schemas/src/credential.ts:94`); `renewCredential` + `renewalWindowSeconds` (`credential-manager.ts:402-429`); heartbeat path exists |
| OAuth token | Long-lived, refresh-rotated | Standard OAuth 2.1; refresh token rotation per spec |

What does *not* exist yet but is small to add:

- `Credential.maxUses` + counter increment on use, for true ephemeral "exactly N messages" credentials. Not needed for v1; just noting the gap.

The natural defaults for the agent-bridge feature:

- **Invite**: 5 min to redeem, single-use (`expiresAfterUse: true`).
- **Credential issued at redemption**: 24h sliding TTL. Renews on heartbeat/use. Dies on (a) idle past renewal window, (b) explicit `xs cred revoke`, or (c) chat closure.
- **OAuth token**: standard refresh-token rotation; durable across all future assistant conversations.

## Platform constraints (research summary)

Full brief lives upstream of this note. Headline points:

- **Transport.** Both platforms require remote MCP over HTTPS. Streamable HTTP recommended (ChatGPT explicitly prefers it; Claude.ai supports both Streamable HTTP and SSE).
- **Auth.** OAuth 2.1 + PKCE + DCR + RFC 8707 mandatory on both. Token = bearer auth on every request, audience-bound to your server.
- **Per-user persistence.** TRUE on both with high confidence. The OAuth token is the durable per-user handle; the connector does not re-auth per assistant conversation. Key your server-side state by your AS's stable `sub`, not by the access token (which rotates).
- **Per-conversation context.** Neither platform reliably exposes a stable assistant-conversation ID to the MCP server. Don't design around it. (`_meta` may carry transport session IDs but those are per-MCP-connection, not per-chat-thread.)
- **Push/streaming.** This is the one real divergence and the one real limitation:
  - **ChatGPT** supports `ui/notifications/tool-result` push into an *open widget*. Live ChatGPT session can react to incoming XMTP messages in near-real-time via widget update.
  - **Claude.ai** is strictly pull today. The assistant only sees new XMTP messages when it calls a tool. SSE is used for transport-level streaming of tool responses, not unsolicited push.
  - **Neither** can wake a dormant assistant conversation. If the user isn't currently chatting with the assistant, the assistant cannot see new XMTP messages until next tool call.

Practical implication: design the contract as **pull-with-cursor** as the source of truth on both platforms. Treat ChatGPT widget push as a UX bonus, not load-bearing.

## MCP tool surface

One MCP server core, one thin platform-specific shell (mostly for ChatGPT widget metadata and `ui/notifications/*` wiring).

Core tools:

- `signet.acceptInvite(url)` — redeem invite, mint credential against the current operator, persist binding. Idempotent on the URL slug (re-redemption is a no-op or returns the existing credential).
- `signet.listChats()` — return active credentials and chat metadata for this operator.
- `signet.recent(chat)` — convenience wrapper. Returns messages newer than the server-side per-(operator, chat) watermark. Does not advance the watermark.
- `signet.poll(chat, afterMsgId?)` — explicit cursor-based read. Returns messages with `nextCursor` (`= latest msg_id in batch`), `hasMore`, and `lagSeconds`. Read-only, idempotent, safe to retry. Default for assistants that want full control.
- `signet.markRead(chat, throughMsgId)` — explicit watermark commit. Side-effecting. Called only when the assistant has actually processed the batch. Decoupling read from acknowledge avoids the retry-skips-messages footgun (relevant: ChatGPT docs say tool calls may be retried, handlers should be idempotent).
- `signet.send(chat, body)` / `signet.reply(chat, msgId, body)` — write surface.
- `signet.subscribe(chat)` — no-op on Claude.ai, registers widget push subscription on ChatGPT.
- `signet.leaveChat(chat)` — connector-side credential self-revocation.

Cursor model details:

- Every message has stable `msg_<16hex>` resource ID. That's the cursor. Server returns `nextCursor` in every read response; client passes it back next time.
- Two cursor types supported: `afterMsgId` (preferred — exact, idempotent, no clock skew) and `since: <ISO8601>` (convenience for "last hour" UX; needs stable secondary sort for same-millisecond messages).
- Server-side watermarks are per-(operator, chat) and only advance via explicit `markRead`. `recent` reads from the watermark without advancing it; `poll` ignores the watermark entirely.

Bonus signal worth surfacing in `poll` responses: `lagSeconds` (how far behind the live tail this batch is) and `hasMore: bool` (more messages waiting beyond this batch). Useful for the assistant to decide "should I summarize the last hour or just respond to the latest message."

## Revocation surface (right granularity at each layer)

- `xs cred revoke cred_X` — kicks the connector out of one chat. Other chats remain.
- `xs cred update cred_X --scope ...` — narrow scope without revoking. E.g., demote a noisy agent to `observation` only.
- `xs operator rm op_a7f3` — kicks the entire connector install off the signet. The lever for "Claude.ai install was compromised."
- Chat-side: `xs chat archive conv_X` (or equivalent) → tombstone honored by credential checker, all credentials for that chat go invalid.

## Open design questions

1. **URL form for the invite.** Custom scheme (`xmtp-signet://`) vs. signet-hosted HTTPS endpoint. Custom scheme is more "drop anywhere" but won't unfurl in most chat clients; HTTPS endpoint unfurls nicely but requires a publicly reachable signet. Maybe both, with the HTTPS form 302-redirecting to the custom scheme on devices that have it registered.
2. **One operator per install or one per (install, chat)?** User said one connector install ≠ one credential, which means one operator per install owning N credentials. Confirmed direction.
3. **Cross-platform operator linking.** Out of scope for v1, but the data model should not preclude it. A simple `link_code` table keyed by operator with 10-min TTL handles this when we add it.
4. **Profile broadcast.** When a credential is minted, should the signet auto-publish a profile message to the chat announcing the new agent member (using the existing `profile-messages` machinery)? Probably yes — it's the existing onboarding scheme convention and surfaces "an agent just joined" in the conversation history.
5. **Push on ChatGPT.** Worth the v1 lift, or ship pull-only on both and add ChatGPT push in a follow-up? Argument for shipping pull-only first: same code path on both platforms, less surface area to debug, widget push is genuinely a UX bonus rather than a correctness requirement.
6. **`signet.poll` rate limiting.** If an assistant calls `poll` in a tight loop, is there a server-side floor? Probably yes, with a 429 + `Retry-After`. Not urgent but worth a placeholder in the handler.
7. **Conversation context for the assistant.** When the assistant calls `signet.recent`, should the response include not just messages but also chat metadata (member list, your operator's profile in this chat, scope summary)? Probably yes on first call and on `lastMetadataChangedAt` invalidation; otherwise it's wasted tokens.

## What's needed beyond what exists today

In rough order of size:

1. **`signet-agent` onboarding scheme.** Mirror of `convos/` with self-contained invite URL and credential-mint redemption. Reuses `invite-crypto.ts`.
2. **MCP transport upgrade.** Confirm the existing `packages/mcp/` surface speaks Streamable HTTP, OAuth 2.1 + PKCE + DCR + RFC 8707 audience binding. If not, that's the lift.
3. **AS for OAuth.** New surface area. Could be a minimal embedded AS in the signet (handles DCR, issues tokens, maps `sub` → operator). Or delegate to a hosted IdP for production. v1 can be the embedded path.
4. **Per-(operator, chat) read watermark store.** Small addition to the sessions package.
5. **`signet.poll` / `signet.recent` / `signet.markRead` handlers.** Straightforward against the existing message machinery.
6. **ChatGPT widget shell.** Optional v1; required if we want widget push.
7. **`signet.listMyChats` handler.** "Given this operator, what chats am I in?" — needed regardless, used by both pull surfaces and widget surfaces.
