## Architectural Decisions (confirmed with user)

1. **ActionSpec handlers live in runtime packages**, not CLI. Each package exports a factory: `createSessionActions(sessionManager) → ActionSpec[]`. CLI is just the composition root that calls factories and registers results. Enables "define once, expose everywhere."

2. **Lazy BrokerCore init**. Keys → sessions → admin socket → WS server start at boot. XMTP client connects lazily on first group join or message operation. Admin path works without network.

3. **Shared admin client helper**: `createAuthenticatedClient(paths, keyManager)` — vault → JWT → Unix socket → auth frame in one call. Every daemon-mode CLI command uses it.

4. **Reference CLIs** (.reference/xmtp-cli, convos-cli) use hardcoded oclif commands — broker's ActionRegistry is a better pattern for multi-transport. Go our own route.

5. **`viem` added as blessed dep** for secp256k1 XMTP identity keys. Signer adapter uses `privateKeyToAccount` + EIP-191 signing. No wallet client or RPC needed.

6. **qmd collections indexed**: `xmtp-broker` (docs), `xmtp-broker-plans` (.agents), `xmtp-broker-claude` (.claude) — 48 files with embeddings. Use `qmd query "..." -c xmtp-broker-plans` for spec lookups.
